# 开源协议说明

本仓库现在采用“双许可证”结构，而不是单一许可证。

原因很简单：它已经不再是纯文档仓库，而是同时包含：

- 攻略内容与说明文字
- JSON 数据层
- HTML / CSS / JavaScript 前端代码
- 校验脚本

因此，内容层和代码层分别使用不同但都带有“保留来源 + 派生继续开源”特征的协议。

## 当前许可证结构

### 内容层：`CC BY-SA 4.0`

适用范围通常包括：

- `content/`
- `docs/data/`
- 维护者原创或依法有权授权的说明性文档内容

它的核心效果是：

- 允许复制、传播、改编和再发布
- 必须保留署名与来源
- 如果你改编后再发布，必须继续使用相同或兼容的 ShareAlike 协议开放

这意味着：

- 别人可以使用你的攻略内容
- 但必须保留来源
- 改编后不能闭源占有

### 代码层：`GPL-3.0-or-later`

适用范围通常包括：

- `docs/*.html`
- `docs/assets/css/`
- `docs/assets/js/`
- `scripts/`

它的核心效果是：

- 允许使用、学习、修改和再分发代码
- 必须保留版权与许可证声明
- 如果别人分发修改后的版本，仍必须继续以 GPL 方式开源

这意味着：

- 别人可以复用你的站点代码
- 但不能把修改后的分发版本闭源
- 也必须保留原有许可信息

## 为什么这样拆分

`CC BY-SA 4.0` 更适合攻略、文章、知识整理、说明文档和结构化内容数据。

`GPL-3.0-or-later` 更适合程序代码和脚本。

如果把整个仓库都只放在一个内容协议下，会让代码部分的授权边界不清楚。
如果把整个仓库都只放在一个代码协议下，又不能很好表达攻略内容的再传播规则。

所以双许可证是这个仓库当前最清晰、最可维护的做法。

## 详细解释：CC BY-SA 4.0

### 允许什么

- 可以转载和分发
- 可以翻译、整理、改写、摘录和二次创作
- 可以在其他项目中继续使用这些内容
- 允许商业使用

### 要求什么

- 必须署名
- 必须保留来源或给出来源链接
- 必须说明你是否做过修改
- 改编后再发布时，必须继续使用同样的 ShareAlike 开放方式

### 不允许什么

- 不允许把改编后的内容改成闭源或更封闭的独占授权再分发
- 不允许移除原始署名与来源信息
- 不允许额外增加技术限制或法律限制，阻止下游继续享有协议赋予的权利

### 对这个仓库意味着什么

- 别人可以转载你的攻略
- 别人可以在此基础上继续整理和扩展
- 但他们必须注明来源，并继续开放他们的改编版本

## 详细解释：GPL-3.0-or-later

### 允许什么

- 可以运行代码
- 可以查看、学习和修改代码
- 可以复制与再分发代码
- 可以发布自己修改后的版本

### 要求什么

- 必须保留版权声明和许可证信息
- 如果分发修改版或派生版，也必须继续使用 GPL-3.0-or-later
- 接收者必须继续享有获取源码、修改源码和再分发源码的权利

### 不允许什么

- 不允许把分发出去的派生版本改成闭源专有版本
- 不允许删除许可证并以独占方式重新发布代码
- 不允许通过额外限制剥夺下游获得源码与再分发的权利

### 对这个仓库意味着什么

- 别人可以拿你的前端页面和脚本继续开发
- 但如果他们分发修改版，也要继续开源这些代码

## 这两种协议分别回答了什么问题

### “别人能不能用？”

可以，内容和代码都可以被使用。

### “是否必须保留来源？”

必须。内容层通过 `BY` 署名要求体现，代码层通过保留版权与许可证声明体现。

### “是否强制继续开源？”

是的。

- 内容层通过 `SA` 实现
- 代码层通过 `GPL` 的 copyleft 实现

## 与之前协议的区别

仓库之前使用的是 `CC BY-NC-SA 4.0`。

现在改成这套双许可证后，最大的变化是：

- 去掉了“非商业使用”限制
- 改为允许他人使用与传播
- 但仍要求保留来源
- 且仍要求派生版本继续开放

换句话说，新的策略更接近“真正的开放共享”，而不是“允许分享但禁止商用”。

## 一个非常重要的边界

本仓库中的协议，只能覆盖维护者依法有权授权的部分。

如果某些材料来自：

- 第三方原帖
- 截图
- 外部图片
- 外部素材
- 仍受原作者版权约束的引用内容

那么这些部分是否可被再次授权，要看维护者是否真的拥有对应授权。

因此，使用本仓库内容时，请同时阅读：
